---
title: "Release discipline"
description: "How Darpan keeps product releases reviewable."
---

Darpan release work is evidence-led. A release should explain what changed, what was verified, and what remains intentionally out of scope.

## Release principles

- Keep release scope small enough to review.
- Tie product changes to documentation updates when user-facing behavior changes.
- Validate the backend engine, UI workflow, and generated outputs that the release touches.
- Promote only reviewed changes into the release branch.
- Record known deferrals separately from shipped behavior.

## What a release note should answer

- Which reconciliation workflows changed?
- Which setup surfaces changed, such as schemas, connections, saved runs, or RuleSets?
- Which backend contracts or generated-output shapes changed?
- Which checks passed before release?
- Are there migration, upgrade-data, or configuration steps?

## Verification

Use the narrowest reliable checks for the changed surface, then broaden when a release touches shared contracts or critical reconciliation flows. Release notes should distinguish between automated checks, manual workflow checks, and unverified areas.
