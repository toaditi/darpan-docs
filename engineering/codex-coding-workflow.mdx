Darpan is developed with Codex as a hands-on engineering partner. The important part is not that an LLM writes code; it is that the work stays grounded in the real application, the real backend contracts, and verifiable behavior.

The product goal stays constant: make it easier for a user to set up, run, and review reconciliation. AI-assisted engineering helps maintain that focus across the Vue app, Moqui services, Spark comparison logic, Drools RuleSets, and documentation.

## What the coding assistant helps with

| Area | Contribution |
| --- | --- |
| Codebase discovery | Traces a product behavior from app route to API client, facade service, entity, backend helper, and test. |
| UI implementation | Keeps custom UI in the Vue app while reusing shared page and workflow patterns. |
| Backend implementation | Keeps service contracts explicit in Moqui XML and places dense processing in testable Groovy or Java helpers. |
| Reconciliation logic | Preserves the split between Spark dataset work and Drools-backed configurable decisioning. |
| Verification | Runs focused checks, reports what passed, and calls out remaining risk. |
| Documentation | Updates product and engineering docs from implemented behavior instead of aspirational architecture. |

## Engineering principles

AI-assisted work on Darpan follows the same standards expected from human engineering work:

- Preserve repository ownership boundaries.
- Prefer the smallest production-quality change that solves the problem.
- Keep user-facing workflows centered on reconciliation runs.
- Keep backend behavior contract-first and Moqui-native.
- Remove weak compatibility layers when a simpler contract is available.
- Treat tests, logs, and runtime behavior as stronger evidence than assumptions.

## Why it matters for Darpan

Darpan is a sophisticated reconciliation application. A single visible workflow can touch saved setup, schema selection, source data, tenant access, JSON-RPC services, Spark comparison, Drools rules, generated output, and result review.

Using Codex well makes that complexity easier to manage. It helps connect the product surface to the backend engine without hiding business behavior in one-off code, and it keeps documentation close to what the application actually does.
