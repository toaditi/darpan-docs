Engineering docs are for Darpan team members and operators. They should stay grounded in the actual Darpan repositories and avoid aspirational architecture that the code does not support yet.

The backend component is the product engine. Start with [Backend engine overview](/backend/overview) when you need to understand service contracts, entities, reconciliation processing, RuleSets, tenant access, or generated outputs.

## Repo boundaries

- [`toaditi/darpan-ui`](https://github.com/toaditi/darpan-ui) is the Vue 3, TypeScript, and Vite frontend app. It owns custom UI, routes, the Ask Darpan shell, browser interaction, and frontend API clients.
- [`toaditi/darpan`](https://github.com/toaditi/darpan) is the Darpan backend Moqui component. In the local Moqui checkout it is mounted at `darpan-backend/runtime/component/darpan`. It owns Moqui XML services, entity definitions, facade contracts, Groovy processing helpers, Spark/Drools reconciliation processing, and component-local backend docs.
- [`toaditi/darpan-docs`](https://github.com/toaditi/darpan-docs) is this Mintlify documentation site. It should document the product and engineering surface, not replace source-level docs inside the UI or backend repos.
- Cross-repo changes are valid when UI behavior needs new backend data or service contracts, but each code change should land in the repo that owns that behavior.

## Workflow rules

- Classify actionable product work as feature, bug fix, or neither.
- Use Linear as the source of truth for feature and bug work.
- Keep GitHub focused on branches, commits, pull requests, and review artifacts.
- Use topic branches for active product code work.

## Documentation rule

When product behavior changes, update the relevant product docs in the same delivery stream or explicitly record why no docs change is needed.
