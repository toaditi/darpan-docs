The Backend engine section explains how reconciliation runs move through Moqui services, Spark processing, Drools RuleSets, tenant access, and generated outputs. This section explains how Darpan is developed on top of that architecture with Codex-assisted engineering.

Darpan uses Codex as a coding partner for code reading, implementation, verification, and documentation updates. The goal is to keep a sophisticated reconciliation application maintainable without losing sight of the product workflow: set up a reconciliation run, execute it, and review the result.

## What agent coding means here

Agent coding is not a separate product feature. It is the engineering practice used to work on Darpan safely:

- Start from the user workflow, especially reconciliation run setup and review.
- Read the existing route, API, service, entity, and processing code before changing behavior.
- Keep custom UI in the Vue app and backend contracts in the Moqui component.
- Use Moqui XML services to keep backend contracts visible and reviewable.
- Use Spark and Drools where dataset comparison or configurable decisioning belongs on the backend.
- Verify the smallest meaningful surface before claiming broader confidence.
- Remove weak compatibility layers when a simpler contract is available.

## Repository boundaries

- [`toaditi/darpan-ui`](https://github.com/toaditi/darpan-ui) is the Vue 3, TypeScript, and Vite frontend app. It owns custom UI, routes, the Ask Darpan shell, browser interaction, and frontend API clients.
- [`toaditi/darpan`](https://github.com/toaditi/darpan) is the Darpan backend Moqui component. In the local Moqui checkout it is mounted at `darpan-backend/runtime/component/darpan`. It owns Moqui XML services, entity definitions, facade contracts, Groovy processing helpers, and Spark/Drools reconciliation processing.
- Cross-repo changes are valid when UI behavior needs new backend data or service contracts, but each code change should land in the repo that owns that behavior.
