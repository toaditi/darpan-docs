This overview explains how Darpan is engineered: a Moqui/Spark/Drools backend, a Vue control surface, and a Codex-assisted coding practice that keeps sophisticated reconciliation workflows maintainable.

Engineering documentation combines backend architecture and agent coding practices in one place. Start with [Backend engine overview](/backend/overview) when you need to understand how a reconciliation run moves through services, entities, Spark processing, RuleSets, tenant access, and generated outputs.

## Repo boundaries

- [`toaditi/darpan-ui`](https://github.com/toaditi/darpan-ui) is the Vue 3, TypeScript, and Vite frontend app. It owns custom UI, routes, the Ask Darpan shell, browser interaction, and frontend API clients.
- [`toaditi/darpan`](https://github.com/toaditi/darpan) is the Darpan backend Moqui component. In the local Moqui checkout it is mounted at `darpan-backend/runtime/component/darpan`. It owns Moqui XML services, entity definitions, facade contracts, Groovy processing helpers, and Spark/Drools reconciliation processing.
- Cross-repo changes are valid when UI behavior needs new backend data or service contracts, but each code change should land in the repo that owns that behavior.

## Agent coding model

Darpan development uses Codex as a coding partner for code reading, implementation, verification, and documentation updates. The strongest results come from a few durable engineering habits:

- Product work starts from the reconciliation-run workflow users actually follow.
- Custom UI stays in `darpan-ui`; backend contracts stay in the Moqui component.
- Moqui XML services keep backend contracts visible and reviewable.
- Spark and Drools are used where dataset comparison or configurable decisioning belongs on the backend.
- Changes are verified against the smallest meaningful surface before broader confidence is claimed.
- Cleanup favors simpler contracts over compatibility layers that make the app harder to reason about.

This workflow matters because Darpan has many moving parts: app shell navigation, schemas, connections, saved runs, generated outputs, tenant scope, backend services, and rule execution. Agent coding works best here when each change is grounded in the current code and verified against the owning surface.
