---
title: "Data access and tenancy"
description: "How the backend scopes users, companies, tenants, and generated files."
---

Darpan backend services must resolve who is acting and which tenant or company scope is active before returning or mutating data.

## Active scope

The backend supports active company or active tenant context in the user session. Product records are scoped so that users see and mutate records for the active scope, not every record they could theoretically access.

Common scoped fields include:

- `ownerUserId`
- `createdByUserId`
- `companyUserGroupId`
- tenant user group IDs on tenant-specific records

## Permission membership

Tenant permission membership is modeled separately from tenant identity. A user can have one active tenant while their permission level for that tenant is resolved from membership records.

The backend entity model includes:

```text
darpan.auth.TenantUserPermissionGroupMember
```

This joins:

- `tenantUserGroupId`
- `userId`
- `permissionUserGroupId`
- effective date fields

That split allows the backend to answer questions such as whether the current user can view or edit the active tenant's data without making one global role apply everywhere.

## Service checks

Facade services should check access before listing, opening, saving, running, or deleting scoped records. The UI should not depend on client-side filtering as the security boundary.

Backend access helpers are responsible for:

- Resolving the active tenant or company
- Checking read or write access
- Applying scope filters to entity queries
- Resolving tenant-scoped temp and output locations

## Generated file isolation

Generated outputs have both database metadata and file-system or object-location data. The backend should resolve output paths from the active tenant scope so that generated files follow the same access model as saved runs and output descriptors.

When documenting generated outputs, include both sides:

- The descriptor record users can list, open, or delete
- The scoped file location where the generated data is stored

## Documentation rule

Do not document data access as a UI convention. Data visibility, mutation access, and output isolation are backend contracts.
