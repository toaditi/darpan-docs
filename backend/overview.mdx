title: "Backend engine overview"
description: "How the Darpan backend is organized and why it is the product engine."
---

Darpan's UI is the control surface for setting up and running reconciliation. The backend component is the engine that makes those runs durable, scoped, repeatable, and reviewable.

The backend implementation is the [`toaditi/darpan`](https://github.com/toaditi/darpan) Moqui component. It is mounted inside a Moqui framework checkout at `<moqui-framework>/runtime/component/darpan`.

## Why this stack

| Platform | What it brings to Darpan |
| --- | --- |
| Moqui | Service contracts, entity persistence, security, JSON-RPC transport, session context, and XML-first orchestration for run setup and output review. |
| Apache Spark | Dataset normalization and comparison for files or records that may grow beyond browser-memory workflows. Spark lets the backend prepare comparable datasets and compute base diffs predictably. |
| Drools/KIE | RuleSet compilation and execution for configurable reconciliation decisions. Drools lets Darpan keep comparison and classification behavior in rules instead of hardcoded branches. |

Together they let Darpan keep the user experience simple while the backend handles validation, access scope, data preparation, rule execution, and generated-output persistence.

## Current shape

- Moqui component descriptor: `component.xml`
- Public service surface: Moqui JSON-RPC services through `/rpc/json`
- Service contracts: XML service definitions under `service/**`
- Data model: Moqui entity definitions under `entity/**`
- Processing helpers: Groovy and Java code under `src/main/groovy/**`
- Reconciliation engine: Spark-backed compare services with RuleSet and legacy mapping paths
- Rule engine: Drools/KIE services for compiling and executing reconciliation rules
- Output handling: generated-output descriptors plus tenant-scoped files

## What the backend owns for a run

The backend owns the durable product behavior:

- It validates user and tenant access before reads and writes.
- It resolves active-company or active-tenant scope for records and generated files.
- It stores schemas, reconciliation runs, RuleSets, compare scopes, connection records, and output descriptors.
- It normalizes uploaded or remote source data into compare-ready datasets.
- It runs base diffs and rule-backed comparisons.
- It returns the JSON-RPC responses the UI renders.

## What the UI should assume

The UI should treat backend facade services as the contract. It should not recreate reconciliation rules, tenant filtering, generated-output paths, or service-side validation in browser code.
